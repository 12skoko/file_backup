     异地文件备份系统 — 设计计划

     Context

     用户需要一个异地备份系统，A 地 → B 地单向同步。核心需求：
     - 智能移动检测（hash 匹配 → B 端 rename，不传文件）
     - B 地已有部分文件但不全，增量补齐
     - Python 负责扫描 + diff，rclone 负责文件传输
     - 先手动运行 Python 调试，将来再 Docker 化

     已确认的设计决策：
     - HTTP API 加 token 认证
     - B 端多余文件移入回收站（.backup_trash），并写入报告
     - 文件排除规则（.backupignore）
     - 支持多对路径映射（A 和 B 路径可以不同）
     - 空目录同步
     - B 端删除走回收站（.backup_trash）
     - 两步流程：diff 报告 → 用户确认 → 执行同步 → 同步报告
     - 暂不处理：文件写入中、双向并发同步、Docker 部署

     工作流程

     backup diff (预览模式)                  backup sync (执行模式)
     ─────────────────────                   ─────────────────────
     扫描 A 本地                              扫描 A 本地
       ↓                                        ↓
     请求 B 扫描任务 ─HTTP→ B 后台扫描         请求 B 扫描任务 ─HTTP→ B 后台扫描
       ↓                                        ↓
     轮询 B 扫描结果                          轮询 B 扫描结果
       ↓                                        ↓
     对比生成 diff 报告                        对比生成 diff 报告
       ↓                                        ↓
     输出报告，程序结束                      输出报告，等待用户确认 (y/N)
                                                ↓ 用户确认
                                             rclone 执行同步
                                                ↓
                                             生成同步报告，程序结束

     架构总览

     两个 Python 进程，同一份代码，分别使用 source.yaml 与 target.yaml：

              A 机器                                    B 机器
        ┌──────────────────────┐            ┌──────────────────────┐
        │  backup diff / sync  │   HTTP     │  backup serve        │
        │  (一次性命令)          │←─────────→│  (常驻后台)            │
        │                      │  +token   │                      │
        │  ① 扫描本地           │           │  ① 扫描本地           │
        │  ② 创建 B 扫描任务     │           │  ② 后台扫描任务队列     │
        │  ③ 轮询 B 扫描结果     │           │  ③ 返回树 JSON        │
        │  ④ diff → 报告        │  WebDAV   │                      │
        │  ⑤ (sync) 确认后执行  │──────────→│  rclone serve webdav │
        │     rclone 操作B      │           │  (暴露各路径供A操作)   │
        └──────────────────────┘            └──────────────────────┘

     A 端：按需执行 backup diff（预览）或 backup sync（执行），跑完退出。定时由外部 cron/任务计划程序触发。

     B 端：backup serve 常驻后台，被动响应 /scan 任务请求 + 暴露 WebDAV。

     各模块设计

     1. config.py — 配置管理

     校验规则：
       - 配置文件按角色拆分：source.yaml 只用于 A 端，target.yaml 只用于 B 端
       - source.yaml 的 paths 至少有一对，source 路径必须存在且为目录
       - target.yaml 的 paths 至少有一对，target 路径必须存在且为目录
       - 两端 paths 使用相同的 name 作为路径映射身份；A 端 source[name] 对应 B 端 target[name]
       - target 模式下，所有 target 路径必须在 webdav.root 下（否则报错）
       - trash.dir 必须在 webdav.root 下（否则报错），因为 A 端通过 WebDAV 执行 moveto
       - HTTP API 端口与 WebDAV 端口必须不同（否则报错）
       - token 不能为空
       - source.yaml 中 target.api_url 必须可连接
       - 多 path 之间有重叠时发出警告
       - 所有路径在加载时归一化：展开 ~，转绝对路径，消除 ..，内部相对路径统一使用 POSIX 分隔符 /

     # source.yaml — A 端配置

     # ──────── HTTP API 认证 ────────
     server:
       token: "your-secret-token-here" # 两边一致

     # ──────── 路径映射 ────────
     # name 是两端路径映射的稳定身份；source 是 A 端本地路径
     paths:
       - name: photos
         source: /mnt/photos
       - name: documents
         source: /mnt/documents

     # ──────── 排除规则 ────────
     exclude:
       file: .backupignore             # 排除规则文件路径
       # 如果文件不存在则用内置默认规则

     # ──────── 缓存 ────────
     cache:
       dir: /data/.backup              # 缓存目录（存放 cache.db）

     # ──────── 对端连接 ────────
     target:
       api_url: http://192.168.1.100:9527
       rclone_remote: B_webdav
       scan_poll_interval_sec: 2
       scan_timeout_sec: 3600

     # ──────── rclone ────────
     rclone:
       binary: rclone
       retries: 3
       transfers: 4

     # ──────── 同步行为 ────────
     sync:
       dry_run: false
       report_dir: /data/.backup       # 报告输出目录

     # target.yaml — B 端配置

     # ──────── HTTP API 服务 ────────
     server:
       host: 0.0.0.0
       port: 9527
       token: "my-secret"

     # ──────── WebDAV 服务 ────────
     webdav:
       host: 0.0.0.0
       port: 9528
       root: /backup                   # 所有 target 路径和 trash.dir 的公共父目录

     # ──────── 路径映射 ────────
     # name 必须与 source.yaml 中对应项一致；target 是 B 端实际落盘路径
     paths:
       - name: photos
         target: /backup/photos
       - name: documents
         target: /backup/docs

     # ──────── 排除规则 ────────
     exclude:
       file: .backupignore

     # ──────── 缓存 ────────
     cache:
       dir: /backup/.backup_cache

     # ──────── 回收站 ────────
     trash:
       dir: /backup/.backup_trash      # 必须在 webdav.root 下
       keep_days: 30

     # ──────── 扫描任务 ────────
     scan:
       max_workers: 1                  # 避免多个全盘扫描同时压垮磁盘
       job_ttl_sec: 3600
       result_ttl_sec: 3600
       graceful_timeout_sec: 10        # 退出时等待扫描 worker/rclone 结束的秒数

     2. scanner.py — 文件 + 目录扫描

     输入: paths 列表 [{name, root_path}], cache_dir, exclude_rules
     输出: ScanResult {
         files: {path_key: {name, rel_path, size, mtime, hash}}, # 所有文件
         dirs:  [{name, rel_path}, ...],                         # 所有目录（含空目录）
     }

     流程:
       遍历每个 {name, root_path}:
         1. os.walk(root_path)
         2. 对每个路径应用排除规则（.backupignore）
         3. 收集所有目录 → dirs 集合
         4. 对每个文件:
            - stat 获取 size, mtime
            - 与 SQLite 缓存对比 → 命中则复用 hash
            - 未命中 → 分块 SHA256，更新缓存
            - rel_path 归一化为 POSIX 风格相对路径
            - path_key = "{name}/{rel_path}"
            - files[path_key] = {name, rel_path, size, mtime, hash}
         5. 返回 {files, dirs}

     SQLite 表结构：
     CREATE TABLE file_cache (
         path       TEXT PRIMARY KEY,
         size       INTEGER,
         mtime      REAL,
         hash       TEXT,
         scanned_at TEXT
     );
     CREATE INDEX idx_hash ON file_cache(hash);

     缓存按 path map 隔离，不同 path pair 的缓存存在同一个 DB 中。
     建议缓存主键使用 (map_name, normalized_abs_path)，避免不同映射中的相同相对路径互相污染。

     路径归一化规则：
       - 本地路径：expanduser → abspath → normpath
       - 内部相对路径：始终使用 /，禁止以 / 开头，禁止包含 ..
       - 路径 key："{map_name}/{rel_path}"
       - rclone/WebDAV 路径：由 target.yaml 的 target 路径相对于 webdav.root 计算得出，再拼接 rel_path
       - Windows 下比较路径时保留原始大小写，但校验重叠路径时使用 normcase

     4. differ.py — 智能对比

     输入: scan_a, scan_b (各含 files + dirs)
     输出: Plan {
         uploads:  [...],     # 需要上传的文件
         moves:    [...],     # B 端移动 (old_path → new_path)
         trashes:  [...],     # B 端移入回收站
         mkdirs:   [...],     # B 端需创建的目录
         extra:    {...},     # B 端多出的文件/目录（文件会进回收站，同时报告）
     }

     文件 diff:
       1. 构建 hash→paths 映射 a_map, b_map
       2. hash 相同、路径相同 → 不变
       3. hash 相同、路径不同 → 候选 Move (在B端 renamed)
       4. A有B无 → Upload
       5. B有A无 → Trash (移到回收站) → 同时记入 extra 报告

     目录 diff:
       1. a_dirs - b_dirs → Mkdir
       2. b_dirs - a_dirs → 如果目录为空或执行后变空，可移入回收站；否则仅记入 extra_dirs，避免误删包含文件的目录

     重复文件处理（同hash多路径）:
       - 不能只按 hash 简单一对一匹配
       - 第一优先级：相同 path_key 直接匹配
       - 第二优先级：同 map_name 内，按目录相似度、文件名相似度、mtime 接近程度打分
       - 第三优先级：跨 map_name 默认不自动 Move，除非显式允许跨映射移动
       - 一个 B 文件只能匹配一个 A 文件，一个 A 文件也只能匹配一个 B 文件
       - 匹配分低于阈值时，保守处理为 Upload + Trash，避免误把不相关副本 rename
       - diff 报告中标记 move 的 confidence，方便用户审阅

     Operation 类型：
     @dataclass
     class MoveOp:     old_path: str; new_path: str
     @dataclass
     class UploadOp:   path: str; local_root: str; target_root: str
     @dataclass
     class TrashOp:    path: str                     # 移入回收站
     @dataclass
     class MkdirOp:    path: str                     # 在B端创建

     5. server.py — HTTP 服务 + 客户端

     B 端常驻 HTTP 服务器，A 端使用同一模块中的客户端函数创建扫描任务并轮询结果。
     B 端扫描可能耗时很长，不能在 HTTP 请求中同步执行，否则会阻塞 HTTP 进程。
     因此 B 端使用异步扫描任务模型：HTTP 请求只负责创建任务、查询状态、取结果；实际 os.walk + hash 在后台 worker 中执行。

     B 端启动:
       1. 读取 target.yaml
       2. 校验所有 target 路径均在 webdav.root 下
       3. 校验 trash.dir 在 webdav.root 下
       4. 校验 server.port != webdav.port
       5. 启动 rclone serve webdav --addr {webdav.host}:{webdav.port} {webdav.root} 子进程
       6. 启动 HTTP server
       7. 启动后台扫描 worker / 线程池
       8. 注册退出清理逻辑：Ctrl+C、SIGTERM、异常退出时关闭 HTTP server、停止扫描 worker，并终止 rclone 子进程

     A 端不需要 server.py —— 直接跑 backup diff / backup sync 命令
     A 端通过 server.start_scan(api_url, token) 创建 B 端扫描任务，再通过 get_scan_status()/get_scan_result() 定时轮询

     所有端点验证 Authorization: Bearer <token>

     端点:
       GET  /health        → {"status": "ok"}
       POST /scan          → 创建扫描任务，返回 {"job_id": "...", "status": "queued"}
       GET  /scan/{job_id} → 查询任务状态：
                             {"job_id": "...", "status": "queued|running|done|failed", "error": null}
       GET  /scan/{job_id}/result
                           → 任务完成后返回 {files: {...}, dirs: [...]}

     扫描任务规则:
       - max_workers 默认 1，避免多个全盘扫描同时运行
       - 如果已有 running/queued 任务，可复用同一个 job_id，或返回 409 让 A 端等待；第一版建议复用最近任务
       - result_ttl_sec 内可重复读取结果，过期后清理
       - 扫描失败时保存错误信息，A 端报告失败而不是无限轮询
       - A 端轮询间隔和超时时间由 source.yaml 的 target.scan_poll_interval_sec / scan_timeout_sec 控制

     B 端进程生命周期:
       - backup serve 是唯一前台进程，HTTP server、扫描 worker、rclone serve webdav 都由它统一管理
       - rclone serve webdav 使用 subprocess.Popen 启动，必须保存进程句柄
       - 收到 KeyboardInterrupt / SIGINT / SIGTERM 时：
         1. 停止接受新的 HTTP 请求
         2. 标记扫描任务取消，等待当前扫描 worker 在 graceful_timeout_sec 内退出
         3. 调用 rclone_proc.terminate()
         4. 在 graceful_timeout_sec 内等待 rclone 退出
         5. 超时仍未退出则 rclone_proc.kill()
       - 如果 HTTP server 或主线程异常退出，也必须进入同一套 cleanup 流程
       - 如果 rclone 子进程提前退出，backup serve 应报告错误并退出，避免 HTTP 还活着但 WebDAV 不可用
       - Windows 下使用 terminate 作为主路径；Linux/macOS 下优先 SIGTERM，再 SIGKILL
       - cleanup 逻辑必须幂等，多次触发也不会重复 kill 或抛异常

     为什么 B 端跑 HTTP 而 A 端不跑：diff/sync 都是 A 主动发起的，A 是客户端，不需要后台服务。B 端 serve 跑着等 A 来连。

     6. executor.py — 执行器

     输入: Plan, paths 映射, config
     输出: 执行结果列表

     执行顺序（重要）:
       1. Trash    —移入回收站 (rclone moveto → trash dir)
       2. Move     —B 端重命名 (rclone moveto)
       3. Upload   —传输新文件 (rclone copy)
       4. Mkdir    —创建空目录 (rclone mkdir)

     回收站逻辑:
       trash_path = {trash.dir}/{timestamp}/{map_name}/{original_relative_path}
       例: /backup/.backup_trash/20260619_143022/photos/old.jpg

       注意：.backup_trash 自动加入排除列表，防止被扫描回传。
       启动时校验 trash.dir 必须在 webdav.root 下，否则 A 端无法通过 WebDAV 执行 moveto。

     对每个 Operation 生成 rclone 命令 (path 均相对于 webdav.root)：
       TrashOp  → rclone moveto {remote}:{rel_path} {remote}:{trash_rel}/{ts}/{rel_path}
       MoveOp   → rclone moveto {remote}:{old_rel} {remote}:{new_rel}
       UploadOp → rclone copy {local_path} {remote}:{rel_target_path}
       MkdirOp  → rclone mkdir {remote}:{rel_path}

     例：webdav.root=/backup, target=/backup/photos
        文件 /backup/photos/2024/me.jpg 在 rclone 命令中的路径为 photos/2024/me.jpg

     7. reporter.py — 报告

     两种报告：diff 报告（执行前）和 sync 报告（执行后）

     diff 报告（backup diff 或 backup sync 执行前输出）：
     {
       "type": "diff",
       "timestamp": "2026-06-19T14:30:22",
       "summary": {
         "will_upload": 5,
         "will_move": 3,
         "will_trash": 2,
         "will_mkdir": 1,
         "extra_on_target_files": 7,
         "extra_on_target_dirs": 2,
         "total_bytes_upload": 1048576
       },
       "uploads": [{"path": "...", "size": 12345}],
       "moves": [{"old": "...", "new": "..."}],
       "trashes": [{"path": "...", "trashed_to": "..."}],
       "mkdirs": ["path/to/new/empty/dir"],
       "extra_files": [{"path": "...", "size": 12345}],
       "extra_dirs": ["path/to/extra/dir"]
     }

     sync 报告（sync 执行后输出，在 diff 报告基础上追加执行结果）：
     {
       "type": "sync",
       "timestamp": "2026-06-19T14:30:22",
       "duration_sec": 12.5,
       "diff": { ... },           // 上面的 diff 报告内容
       "results": {
         "uploaded": [{"path": "...", "status": "ok", "speed": "10MB/s"}],
         "moved": [{"old": "...", "new": "...", "status": "ok"}],
         "trashed": [{"path": "...", "trashed_to": "...", "status": "ok"}],
         "mkdirs": [{"path": "...", "status": "ok"}],
         "errors": []
       }
     }

     报告保存到 sync.report_dir，文件名如 diff_20260619_143022.json / sync_20260619_143022.json。

     8. cli.py — 命令行

     backup serve             启动 B 端常驻服务
       --config target.yaml

     backup diff              扫描 + diff，输出报告，不执行同步
       --config source.yaml   （从 source.yaml 读取 target.api_url 连接 B 端）

     backup sync              扫描 + diff → 输出报告 → 确认 → 执行 → 输出同步报告
       --config source.yaml
       --yes / -y             跳过确认，直接执行

     同步流程时序

     backup diff                                backup sync
     ───────────                                ───────────
     1. 扫描本地 paths                          1. 扫描本地 paths
     2. HTTP POST B/scan ──→ 创建扫描任务       2. HTTP POST B/scan ──→ 创建扫描任务
     3. 轮询 B/scan/{job_id}                    3. 轮询 B/scan/{job_id}
     4. GET B/scan/{job_id}/result              4. GET B/scan/{job_id}/result
     5. differ.compare()                        5. differ.compare()
     6. 输出 diff 报告                          6. 输出 diff 报告
     7. 退出                                    7. 等待用户输入 y/N
                                                   ↓ 用户确认
                                                8. executor.run(Plan)
                                                9. 输出 sync 报告
                                                10. 退出

     ## 边界情况

     1. **大文件哈希**：64KB 分块读取。mtime+size 缓存命中则跳过。
     2. **符号链接**：跳过。
     3. **网络中断**：rclone 自带 `--retries`。executor 捕获异常并报告。
     4. **重复文件（同hash多路径）**：不能只按 hash 简单匹配；按相同路径、路径相似度、mtime 等综合打分，低置信度时保守兜底。
     5. **回收站目录**：自动加入排除列表，防止被扫描回传。
     6. **多 path 对之间有重叠**：配置文件校验时检查警告。
     7. **空目录**：记录在 dirs 集合中，diff 后在 B 端创建。
     8. **A/B 路径不同**：用 name 绑定 source.yaml 与 target.yaml 中的路径映射。
     9. **路径归一化**：本地绝对路径和内部相对路径分别归一化，避免 Windows/Linux 分隔符、大小写和 `..` 带来的误判。
     10. **B 端扫描耗时**：扫描通过异步任务执行，HTTP 请求只创建任务和查询结果，A 端按配置轮询，避免阻塞整个 HTTP 服务。
     11. **WebDAV 与 HTTP 端口**：必须分开配置，启动时校验不能相同。
     12. **回收站可达性**：trash.dir 必须在 webdav.root 下，否则启动失败。
     13. **后台进程清理**：backup serve 收到 Ctrl+C、SIGTERM 或异常退出时，必须关闭 HTTP server、停止扫描 worker，并 terminate/kill rclone 子进程。

     ## 验证方法

     1. differ 单元测试：mock 数据验证增/删/移/空目录/多余报告
     2. server 单元测试：POST /scan 返回 job_id，后台任务完成后可轮询并读取结果；扫描失败能返回 failed
     3. config 单元测试：HTTP/WebDAV 端口冲突、trash.dir 不在 webdav.root 下、paths name 不匹配等情况能报错
     4. 生命周期测试：启动 backup serve 后发送 Ctrl+C/SIGTERM，确认 HTTP server 退出、扫描 worker 停止、rclone 子进程不存在；模拟 rclone 提前退出时 backup serve 能报错退出
     5. 集成测试：B 端 `backup serve --config target.yaml`，A 端 `backup diff --config source.yaml` 查看预览，`backup sync` 执行同步
     6. 端到端：A 端增/删/移文件，运行 sync，检查 B 端结果、回收站、报告文件
