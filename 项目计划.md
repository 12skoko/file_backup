     异地文件备份系统 — 设计计划

     Context

     用户需要一个异地备份系统，A 地 → B 地单向同步。核心需求：
     - 智能移动检测（hash 匹配 → B 端 rename，不传文件）
     - B 地已有部分文件但不全，增量补齐
     - Python 负责扫描 + diff，rclone 负责文件传输
     - 先手动运行 Python 调试，将来再 Docker 化

     已确认的设计决策：
     - HTTP API 加 token 认证
     - B 端多余文件不删，输出到报告
     - 文件排除规则（.backupignore）
     - 支持多对路径映射（A 和 B 路径可以不同）
     - 空目录同步
     - B 端删除走回收站（.backup_trash）
     - 两步流程：diff 报告 → 用户确认 → 执行同步 → 同步报告
     - 暂不处理：文件写入中、双向并发同步、Docker 部署

     工作流程

     backup diff (预览模式)                  backup sync (执行模式)
     ─────────────────────                   ─────────────────────
     扫描 A 本地                              扫描 A 本地
       ↓                                        ↓
     请求 B 树 ────HTTP──→ 扫描 B 本地         请求 B 树 ────HTTP──→ 扫描 B 本地
       ↓                                        ↓
     对比生成 diff 报告                        对比生成 diff 报告
       ↓                                        ↓
     输出报告，程序结束                      输出报告，等待用户确认 (y/N)
                                                ↓ 用户确认
                                             rclone 执行同步
                                                ↓
                                             生成同步报告，程序结束

     架构总览

     两个 Python 进程，同一份代码，config.yaml 区分角色：

              A 机器                                    B 机器
        ┌──────────────────────┐            ┌──────────────────────┐
        │  backup diff / sync  │   HTTP     │  backup serve        │
        │  (一次性命令)          │←─────────→│  (常驻后台)            │
        │                      │  +token   │                      │
        │  ① 扫描本地           │           │  ① 扫描本地           │
        │  ② 拉 B 树            │           │  ② 返回树 JSON        │
        │  ③ diff → 报告        │  WebDAV   │                      │
        │  ④ (sync) 确认后执行  │──────────→│  rclone serve webdav │
        │     rclone 操作B      │           │  (暴露各路径供A操作)   │
        └──────────────────────┘            └──────────────────────┘

     A 端：按需执行 backup diff（预览）或 backup sync（执行），跑完退出。定时由外部 cron/任务计划程序触发。

     B 端：backup serve 常驻后台，被动响应 /tree 请求 + 暴露 WebDAV。

     各模块设计

     1. config.py — 配置管理

     校验规则：
       - role 必须是 source 或 target
       - paths 至少有一对，source/target 路径必须存在且为目录
       - target 模式下，所有 target 路径必须在 webdav.root 下（否则报错）
       - token 不能为空
       - target.url 在 source 模式下必须可连接
       - 多 path 之间有重叠时发出警告

     # config.yaml — 一切配置的唯一起源

     # ──────── 运行模式 ────────
     role: source                      # source | target
     server:
       port: 9527
       token: "your-secret-token-here" # 两边一致

     # ──────── 路径映射 ────────
     # 支持多对路径，A/B 可以不一样
     paths:
       - source: /mnt/photos           # A 端的路径
         target: /backup/photos        # B 端的路径
       - source: /mnt/documents
         target: /backup/docs

     # ──────── 排除规则 ────────
     exclude:
       file: .backupignore             # 排除规则文件路径
       # 如果文件不存在则用内置默认规则

     # ──────── 缓存 ────────
     cache:
       dir: /data/.backup              # 缓存目录（存放 cache.db）

     # ──────── 回收站 ────────
     trash:
       dir: /data/.backup_trash        # B 端删除前移到此目录
       keep_days: 30                   # 保留天数，0=永久保留

     # ──────── WebDAV（target 模式）────
     webdav:
       root: /backup                     # WebDAV 服务根目录（所有 target 路径的公共父目录）

     # ──────── 对端连接（source 模式）────
     target:
       url: http://192.168.1.100:9527
       rclone_remote: B_webdav

     # ──────── rclone ────────
     rclone:
       binary: rclone
       retries: 3
       transfers: 4

     # ──────── 同步行为 ────────
     sync:
       dry_run: false
       report_dir: /data/.backup       # 报告输出目录

     A 端最小配置：
     role: source
     server:
       port: 9527
       token: "my-secret"
     paths:
       - source: /mnt/photos
         target: /backup/photos
     target:
       url: http://backup-b:9527
       rclone_remote: B_webdav

     B 端最小配置：

     server:
       port: 9527
       token: "my-secret"
     webdav:
       root: /backup                     # rclone serve webdav 暴露此目录
     paths:
       - source: /backup/photos        # target 模式下 source 字段作为本地扫描路径
         target: /backup/photos
     trash:
       dir: /data/.backup_trash

     2. scanner.py — 文件 + 目录扫描

     输入: paths 列表 [(source_path, target_path), ...], cache_dir, exclude_rules
     输出: ScanResult {
         files: {path: {size, mtime, hash}},    # 所有文件
         dirs:  ["path/to/dir", ...],           # 所有目录（含空目录）
     }

     流程:
       遍历每个 (source_path, target_path):
         1. os.walk(source_path)
         2. 对每个路径应用排除规则（.backupignore）
         3. 收集所有目录 → dirs 集合
         4. 对每个文件:
            - stat 获取 size, mtime
            - 与 SQLite 缓存对比 → 命中则复用 hash
            - 未命中 → 分块 SHA256，更新缓存
            - files[relative_path] = {size, mtime, hash}
         5. 返回 {files, dirs}

     SQLite 表结构：
     CREATE TABLE file_cache (
         path       TEXT PRIMARY KEY,
         size       INTEGER,
         mtime      REAL,
         hash       TEXT,
         scanned_at TEXT
     );
     CREATE INDEX idx_hash ON file_cache(hash);

     缓存按 path_map 隔离，不同 path pair 的缓存存在同一个 DB 中（path 用 source_path 的为基准）。

     3. .backupignore — 排除规则

     # 注释
     .git/
     node_modules/
     *.tmp
     *.log
     .DS_Store
     Thumbs.db
     ~$*
     *.part          # 未完成的下载
     *.crdownload

     内置默认规则（即使没有 .backupignore 也生效）：
     .git/
     .DS_Store
     Thumbs.db
     ~$*

     4. differ.py — 智能对比

     输入: scan_a, scan_b (各含 files + dirs)
     输出: Plan {
         uploads:  [...],     # 需要上传的文件
         moves:    [...],     # B 端移动 (old_path → new_path)
         trashes:  [...],     # B 端移入回收站
         mkdirs:   [...],     # B 端需创建的目录
         extra:    {...},     # B 端多出的文件（不操作，仅报告）
     }

     文件 diff:
       1. 构建 hash→path 映射 a_map, b_map
       2. hash 相同、路径相同 → 不变
       3. hash 相同、路径不同 → Move (在B端 renamed)
       4. A有B无 → Upload
       5. B有A无 → Trash (移到回收站) → 同时记入 extra 报告

     目录 diff:
       1. a_dirs - b_dirs → Mkdir
       2. b_dirs - a_dirs → 不删除，记入 extra 报告

     重复文件处理（同hash多路径）:
       - 优先按路径相似度匹配
       - 无法匹配的保守处理为新增+回收

     Operation 类型：
     @dataclass
     class MoveOp:     old_path: str; new_path: str
     @dataclass
     class UploadOp:   path: str; local_root: str; target_root: str
     @dataclass
     class TrashOp:    path: str                     # 移入回收站
     @dataclass
     class MkdirOp:    path: str                     # 在B端创建

     5. server.py — HTTP 服务 + 客户端

     B 端常驻 HTTP 服务器，A 端使用同一模块中的 get_tree() 客户端函数主动拉取 B 端文件树。

     B 端启动:
       1. 读取 config.yaml（role=target）
       2. 校验所有 target 路径均在 webdav.root 下
       3. 启动 rclone serve webdav --addr :{port} {webdav.root} 子进程
       4. 启动 HTTP server

     A 端不需要 server.py —— 直接跑 backup diff / backup sync 命令
     A 端通过 server.get_tree(url, token) 发起 HTTP 请求拉取 B 端文件树

     所有端点验证 Authorization: Bearer <token>

     端点:
       GET  /health        → {"status": "ok"}
       GET  /tree          → 扫描本地 paths，返回 {files: {...}, dirs: [...]}

     为什么 B 端跑 HTTP 而 A 端不跑：diff/sync 都是 A 主动发起的，A 是客户端，不需要后台服务。B 端 serve 跑着等 A 来连。

     6. executor.py — 执行器

     输入: Plan, paths 映射, config
     输出: 执行结果列表

     执行顺序（重要）:
       1. Trash    —移入回收站 (rclone moveto → trash dir)
       2. Move     —B 端重命名 (rclone moveto)
       3. Upload   —传输新文件 (rclone copy)
       4. Mkdir    —创建空目录 (rclone mkdir)

     回收站逻辑:
       trash_path = {trash.dir}/{timestamp}/{original_relative_path}
       例: /data/.backup_trash/20260619_143022/photos/old.jpg

       注意：.backup_trash 自动加入排除列表，防止被扫描回传。

     对每个 Operation 生成 rclone 命令 (path 均相对于 webdav.root)：
       TrashOp  → rclone moveto {remote}:{rel_path} {remote}:{trash_rel}/{ts}/{rel_path}
       MoveOp   → rclone moveto {remote}:{old_rel} {remote}:{new_rel}
       UploadOp → rclone copy {local_path} {remote}:{rel_target_path}
       MkdirOp  → rclone mkdir {remote}:{rel_path}

     例：webdav.root=/backup, target=/backup/photos
        文件 /backup/photos/2024/me.jpg 在 rclone 命令中的路径为 photos/2024/me.jpg

     7. reporter.py — 报告

     两种报告：diff 报告（执行前）和 sync 报告（执行后）

     diff 报告（backup diff 或 backup sync 执行前输出）：
     {
       "type": "diff",
       "timestamp": "2026-06-19T14:30:22",
       "summary": {
         "will_upload": 5,
         "will_move": 3,
         "will_trash": 2,
         "will_mkdir": 1,
         "extra_on_target_files": 7,
         "extra_on_target_dirs": 2,
         "total_bytes_upload": 1048576
       },
       "uploads": [{"path": "...", "size": 12345}],
       "moves": [{"old": "...", "new": "..."}],
       "trashes": [{"path": "...", "trashed_to": "..."}],
       "mkdirs": ["path/to/new/empty/dir"],
       "extra_files": [{"path": "...", "size": 12345}],
       "extra_dirs": ["path/to/extra/dir"]
     }

     sync 报告（sync 执行后输出，在 diff 报告基础上追加执行结果）：
     {
       "type": "sync",
       "timestamp": "2026-06-19T14:30:22",
       "duration_sec": 12.5,
       "diff": { ... },           // 上面的 diff 报告内容
       "results": {
         "uploaded": [{"path": "...", "status": "ok", "speed": "10MB/s"}],
         "moved": [{"old": "...", "new": "...", "status": "ok"}],
         "trashed": [{"path": "...", "trashed_to": "...", "status": "ok"}],
         "mkdirs": [{"path": "...", "status": "ok"}],
         "errors": []
       }
     }

     报告保存到 sync.report_dir，文件名如 diff_20260619_143022.json / sync_20260619_143022.json。

     8. cli.py — 命令行

     backup serve             启动 B 端常驻服务
       --config config.yaml

     backup diff              扫描 + diff，输出报告，不执行同步
       --config config.yaml   （从 config 读取 target url 连接 B 端）

     backup sync              扫描 + diff → 输出报告 → 确认 → 执行 → 输出同步报告
       --config config.yaml
       --yes / -y             跳过确认，直接执行

     同步流程时序

     backup diff                                backup sync
     ───────────                                ───────────
     1. 扫描本地 paths                          1. 扫描本地 paths
     2. HTTP GET B/tree ────→ 扫描 B paths      2. HTTP GET B/tree ────→ 扫描 B paths
     3. differ.compare()                        3. differ.compare()
     4. 输出 diff 报告                          4. 输出 diff 报告
     5. 退出                                    5. 等待用户输入 y/N
                                                   ↓ 用户确认
                                                6. executor.run(Plan)
                                                7. 输出 sync 报告
                                                8. 退出
     7. reporter.write(report)
     8. 完成

     ## 边界情况

     1. **大文件哈希**：64KB 分块读取。mtime+size 缓存命中则跳过。
     2. **符号链接**：跳过。
     3. **网络中断**：rclone 自带 `--retries`。executor 捕获异常并报告。
     4. **重复文件（同hash多路径）**：按路径相似度匹配，保守兜底。
     5. **回收站目录**：自动加入排除列表，防止被扫描回传。
     6. **多 path 对之间有重叠**：配置文件校验时检查警告。
     7. **空目录**：记录在 dirs 集合中，diff 后在 B 端创建。
     8. **A/B 路径不同**：每对 path 独立映射 source→target。

     ## 验证方法

     1. differ 单元测试：mock 数据验证增/删/移/空目录/多余报告
     2. 集成测试：B 端 `backup serve`，A 端 `backup diff` 查看预览，`backup sync` 执行同步
     3. 端到端：A 端增/删/移文件，运行 sync，检查 B 端结果、回收站、报告文件
